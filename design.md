# 设计说明

## 1. 目标与设计原则

系统优化两个彼此关联但不能混为一谈的目标：硬标签正确率，以及规则语义对齐。它不把公开样例当成静态答案库，而把样例视作校准业务口径的材料。正式推理只接收当前文档、数据集类型和 intent，防止样例标签泄漏诱导模型过拟合。

## 2. 两类数据集的处理

统一入口先把 PDF、DOCX、文本或批量结构化文件标准化为 `(id, text)`。`dataset_type` 作为显式上下文传入各阶段：

- 计划任务书：侧重目标分解、任务指标、技术路线、进度、资源与验收可执行性。
- 立项申请书：侧重必要性、创新性、可行性、预期价值、风险、团队与投入合理性。

这些是检索方向而不是写死的规则；最终规则由当前 intent 和证据动态生成，因此能处理未见过的判断问题。

## 3. 智能体分工

系统采用受控的三阶段 spec-handoff，各阶段只传递结构化 JSON：

1. **证据抽取器**：只抽事实、缺失项和矛盾，不允许输出标签。
2. **规则规划器**：结合数据集类型、intent、业务常识和证据包，提出 3–7 条语义规则，并标记 blocking（否决项）。
3. **独立裁决器**：逐条核验证据，输出命中规则和硬标签；证据不足时降低置信度并触发人工复核标记。

这种上下文压缩只交接事实与规则，不反复携带整篇推理历史，降低注意力污染和自我强化偏差。

## 4. 判断规则定位

规则不是从关键词直接映射，而由 `intent → 判定维度 → 所需证据 → 文档证据` 逐级对齐。规则 ID 只在单次裁决内稳定；`rule_name` 保持可跨样本复用的业务语义，`evidence` 必须是原文可定位的短片段。缺失和反证分开处理：普通材料未提及不自动等于不通过，只有明确的必备项缺失或否决规则触发才可直接否决。

如果赛方公开训练集可用，推荐在离线校准阶段聚类同义规则，形成版本化的规则词典及少量反例，但不将完整训练集在每次推理时暴露给智能体。词典只作为候选召回源，仍须以当前原文证据确认命中。

## 5. 硬标签输出

裁决顺序为：

1. 检查明确触发的 blocking 规则；命中则优先“不通过”。
2. 核验核心规则是否有足够、相互一致的证据。
3. 综合支持与反对证据，输出唯一硬标签。
4. 同时输出置信度和 `needs_review`，但不以“未知”替代评测要求的硬标签。

高度不平衡通常源于业务准入流程：提交材料经过预筛，真实“通过”可能占多数；或评测专门收集缺陷案例导致反向偏斜。系统不假定多数类，也不按标签先验强行纠偏。离线开发应使用 balanced accuracy、macro-F1、每类召回率和混淆矩阵，并在开发集上校准阈值。

## 6. 泛化能力

- **意图触发**：不依赖固定意图枚举，原始 intent 是所有阶段的一等输入。
- **语义规则**：规则由判定维度和证据需求描述，不记忆样本措辞。
- **证据约束**：结论必须回指当前文档，减少世界知识幻觉。
- **阶段隔离**：抽取器不知道标签，抑制围绕预期结论选择事实。
- **低温结构化输出**：固定 schema、`temperature=0`，提高复现性。
- **错误显式化**：单条失败不会让整个批次失败；输出对应 error。

## 7. Research loop 与后续增强

当前提交采用模型世界知识完成规则发现。生产版可加受控 research loop：仅当规则含法规、行业标准或时效性事实时，检索权威来源，记录来源版本后再交给裁决器。长期记忆只保存经过人工确认的规则定义、同义词和校准统计，不保存隐藏评测文档或未经核验的模型结论。

建议后续加入：OCR、长文档分块检索与章节坐标、基于公开训练集的规则簇评测、双模型仲裁、Prompt 版本追踪、脱敏审计日志和 API 限流。

## 8. 工程结构

```text
app/parser.py   文档解析与批量拆分
app/agent.py    三阶段智能体与降级策略
app/schemas.py  结构化输入输出契约
app/main.py     FastAPI、并发控制、错误隔离
static/         评测网页
tests/          基础回归测试
```
